# 系统架构

## 目标与非目标

系统目标是在尽量低的数据成本下，尽早发现 Web3 事件，把可验证事实、市场影响和来源证据加工成适合不同币安广场账号的内容，并以可审核、可撤回、可追踪的方式分发。

当前不追求全网全量采集、无人值守发布所有类型新闻，也不把社交热度等同于事实。

## 领域模型

| 对象 | 作用 | 关键字段 |
|---|---|---|
| Source | 数据源及预算约束 | 类型、可信度、频率、成本等级 |
| Raw item | 保留原始输入 | 原文、URL、作者、采集时间、内容哈希 |
| Event | 去重聚类后的现实事件 | 实体、主题、首次发现、状态、热度 |
| Evidence | 支持或反驳事件的证据 | 来源等级、摘录、截图、时间戳 |
| Fact pack | 写作唯一事实边界 | 已确认事实、未确认说法、市场反应 |
| Content draft | 针对账号的二创版本 | 角度、语气、引用、风险标签 |
| Asset | 视觉素材 | 证据截图、数据图、信息卡及哈希 |
| Publication | 一次平台发布意图 | 账号、状态、幂等键、外部帖子 ID |
| Audit log | 不可省略的操作记录 | 操作者、动作、前后状态、时间 |

## 分层

1. **采集层**：适配 RSS、公开 API、网页读取、WebSocket、链上 RPC 和付费增强源。每个适配器统一输出 Raw item，并通过诊断层选择当前健康后端。
2. **情报层**：标准化实体和时间，按 URL、内容哈希、标题相似度和实体重合聚类；计算热度与置信度。
3. **编辑层**：构建 Fact pack，路由到最合适账号，按模板生成文案和最多四张视觉素材。
4. **治理层**：检查来源、敏感类型、账号频率、重复发布、投资建议和版权风险。
5. **分发层**：将批准内容放入 Outbox，由平台适配器幂等发布，保存响应并处理重试。
6. **反馈层**：采集曝光、互动、关注转化、纠错与撤稿，反向调整选题和路由权重。

## 状态机

```text
detected -> clustered -> verified -> drafted -> review -> approved -> queued
    |           |          |          |          |                    |
    v           v          v          v          v                    v
 ignored      merged     disputed   rejected   revision            published
                                                                    |
                                                                    v
                                                             failed / withdrawn
```

关键约束：

- `verified` 只表示证据达到内部阈值，不代表未来不会出现更正。
- `approved` 必须记录审核人或明确的自动化规则版本。
- `published` 必须同时保存平台返回 ID 和本地幂等键。
- 更正不能覆盖历史记录，应新增版本并链接到原发布。

## 热点评分

默认权重定义在 `config/policies.example.json`：

```text
score =
  velocity × 25%
  + market anomaly × 20%
  + source trust × 20%
  + cross-source agreement × 15%
  + asset relevance × 10%
  + novelty × 10%
```

所有输入先归一化到 `0..100`，输出也限制在 `0..100`。评分用于排序，不替代事实核验。来源可信度和事件置信度是两个不同字段：可信来源也可能只是在转述未经证实的信息。

## 去重和多账号协同

去重顺序：

1. 规范化 URL 相同，直接合并。
2. 内容哈希相同，直接合并。
3. 24 小时窗口内，标题相似且核心实体高度重合，进入同一事件。
4. 语义相近但时间、数额或主体冲突，保留为争议证据，禁止静默合并事实。

默认一个事件只有一个主账号。其他账号只有满足以下条件才可跟进：角度不同、有新增证据、距离主发布达到冷却时间，且文案与配图不是简单替换措辞。

## 安全边界

- 采集器只持有各自最小权限；发布器不能访问交易 API 或资产私钥。
- Square Key 逐账号隔离，任何一个泄漏都不影响其他账号。
- UI、日志和错误消息永不回显完整 Key。
- 外部抓取内容视为不可信输入，禁止其中的提示语控制模型或系统工具。
- 文案生成输入只包含经过转义的 Fact pack，不把网页原文拼接进系统指令。
- 对发布请求使用 outbox + 幂等键，避免超时重试导致重复发帖。

## 可扩展接口

新增数据源只实现 `fetch -> normalize -> checkpoint`；新增发布平台只实现 `validate -> upload assets -> publish -> lookup result`。领域评分、事实包和内容路由不依赖具体平台，以便未来扩展 Telegram、网站、邮件或其他内容平台。

来源接入额外遵循 `ordered backends -> real probe -> active backend -> remediation`：候选顺序代表偏好，不代表可用；只有真实探测成功的后端才能进入 active 状态。诊断层对结果做短缓存，并把单源异常转换为结构化状态，避免故障扩散到整个控制台。
