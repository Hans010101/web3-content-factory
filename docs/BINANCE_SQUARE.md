# Binance Square 多账号发布方案

## 账号矩阵

推荐先使用四个经合法创建或明确授权的创作者账号：快讯、市场、链上、研究。每个账号有独立主题、语气、最低置信度、每日上限和人工审核规则，模板见 `config/accounts.example.json`。

同一热点默认只由最匹配的账号首发。其他账号需要提供市场、链上、安全或基本面中的独立增量，禁止同步复制刷屏。

## 密钥隔离

每个账号的配置只引用环境变量名：

```json
{
  "id": "market",
  "secretEnv": "BINANCE_SQUARE_API_KEY_MARKET"
}
```

真实 Key 由本地 `.env.local` 或云密钥管理服务注入。正式环境要求：

- 一账号一 Key，不共用、不写入数据库、不进入前端 bundle。
- 发布 Worker 按账号读取对应 Key，其他服务无权读取。
- 日志只显示 Key 指纹，例如末四位；错误响应先脱敏。
- Key 轮换时保留短暂双版本窗口，验证后撤销旧 Key。
- 怀疑泄漏时立即熔断该账号发布并轮换，不影响采集和其他账号。

发布权限必须与交易、提现、钱包私钥完全隔离。

## 发布适配器契约

适配器输入应为平台无关对象：

```json
{
  "publicationId": "pub_01...",
  "accountId": "market",
  "format": "image-post",
  "title": "可选标题",
  "text": "正文",
  "assets": ["asset_01", "asset_02"],
  "sourceLinks": ["https://..."],
  "idempotencyKey": "event-version-account-angle"
}
```

执行顺序：

1. 检查状态为 `approved`、Key 存在且 `PUBLISH_MODE=live`。
2. 验证长度、图片数量、格式、敏感词、来源和账号日配额。
3. 上传图片/视频，保存平台素材 ID；部分成功时可安全续传。
4. 发布正文，保存响应、外部帖子 ID、链接和发布时间。
5. 超时先查询本地/平台结果；无法确认时转人工，不盲目重试。

本地默认 `dry-run`：生成完整请求预览和审计日志，但不调用平台。

当前 `POST /api/publications/run` 是发布 Worker 的本地触发入口：它只处理已审批、已入队且到达计划时间的任务，并通过 Mock Binance Square 适配器返回模拟帖子 ID。后续接入定时队列时保留同一领域服务，HTTP 入口仅用于人工触发、测试和故障补偿；切换真实适配器前必须同时满足 `PUBLISH_MODE=live` 与部署环境密钥检查。

## 幂等、重试和频控

幂等键由 `事件 ID + 事实版本 + 账号 ID + 内容角度` 生成，并在数据库设唯一约束。发布失败按 1、5、20 分钟退避，最多三次；认证错误、内容拒绝和配额错误不自动重试。

同时执行三层限制：

- 平台公开限制：以申请账号时的最新官方文档为准。
- 内部账号日上限：通常远低于平台上限。
- 账号冷却时间：防止突发时连续刷屏。

## 审核等级

| 内容 | 策略 |
|---|---|
| 纯客观市场异动，规则和数据均通过 | 可配置自动发布 |
| 官方常规公告摘要 | 快速人工确认 |
| 监管、上/下币、攻击、破产、冻结 | 强制人工审核 |
| 匿名爆料、内幕、价格目标、买卖建议 | 禁止自动发布 |
| 项目研究和长文 | 全量人工审核 |

审核页至少展示事实包、原始链接、证据截图、市场窗口、文案版本差异、目标账号和发布预览。

## 更正和撤稿

平台适配能力可能不包含编辑或删除，因此系统必须自建：

- 更正队列：生成一条明确链接原帖的更正内容。
- 撤稿记录：即使需要人工去平台删除，也要记录操作者和结果。
- 事件版本：旧事实不覆盖，新证据形成新版本。
- 紧急停发：全局开关和逐账号开关。

正式接入时用小号/测试账号完成文本、1–4 图、长文、中文标点、超时和限流测试，再启用主账号。
