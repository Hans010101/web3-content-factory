# 后续部署步骤

服务器与账号申请暂缓时，先以本地 `dry-run` 完成业务验收。正式上线按以下闸门推进。

## 1. 数据源验收

- 建立官方域名、Feed、仓库、合约和账号白名单。
- 为每个来源记录条款、限流、预估成本、缓存许可和负责人。
- 用历史样本回放，校准重复率、时延和误报。
- 申请 X 等付费源后先设置硬预算，再开放增强采集。

## 2. 基础设施选型

当前应用适合继续部署到 Cloudflare Sites/Workers，并使用 D1/R2；如采集、浏览器截图或队列任务超过 Workers 限制，可拆出 Node Worker 服务。

正式环境至少需要：

- 事务数据库：事件、事实包、审核、发布 outbox 和审计。
- 对象存储：原始响应、截图、图表和成品图片。
- 队列：采集、增强、截图、生成、审核通知和发布。
- 定时器：来源轮询、预算结算、过期事件和指标汇总。
- 密钥管理：按服务和账号最小授权。
- 可观测性：结构化日志、指标、追踪和报警。

## 3. 环境划分

建立 `development`、`staging`、`production` 三套环境，数据库、存储桶、队列和 Key 完全隔离。Staging 只连接测试/专用发布账号，Production 初始仍为 `dry-run`。

部署顺序：数据库迁移 → 只读采集器 → 情报处理 → 截图/生成 → 审核 UI → dry-run 发布器 → 测试账号 live → 主账号灰度。

### 用户身份与个人密钥

- 托管层必须向应用提供已验证的 `oai-authenticated-user-email` 请求头；应用只信任服务端注入的身份，不接受客户端提交 user id。
- 生产环境必须通过平台 Secret 管理设置 `MASTER_ENCRYPTION_KEY`，不要写入仓库、D1 或普通环境配置文件。
- D1 迁移会创建 `user_integrations` 和 `user_ai_preferences`；所有查询均以不可逆哈希后的用户身份作为分区键。
- 若生产环境缺少主加密密钥，系统会拒绝保存 API Key，不会退化为明文或临时密钥。
- 密钥轮换前应先设计旧密钥解密与重加密流程；当前版本不支持直接替换主密钥后读取历史密文。

## 4. 上线闸门

满足以下条件才切换 `PUBLISH_MODE=live`：

- 连续七天采集运行，未出现不可解释的数据缺口或成本超限。
- 事件去重、账号路由和素材数字一致性通过回放测试。
- 高风险内容强制人工审核无法被 API 或 UI 绕过。
- 幂等、超时、429、5xx、部分上传和断网演练通过。
- 全局停发、逐账号停发、Key 轮换和更正流程已演练。
- 完成平台条款、版权、隐私和金融内容合规审查。

## 5. 灰度策略

第一周每天最多 5 条且全人工审核；随后按账号分别放量。市场客观数据可以最先自动化，快讯和链上安全保持人工确认，研究长文长期全审。

核心验收指标：比主流中文来源的领先时间、发布正确率、纠错/撤稿率、单位有效内容成本、关注转化，而不是单纯发帖数量。

## 6. 备份与恢复

每日备份业务数据库，发布与审核审计采用更长保留期；对象存储开启版本或不可变策略。季度演练从备份恢复事件、草稿、素材和 outbox，并验证不会在恢复后重复发布。
